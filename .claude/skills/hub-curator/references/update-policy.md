# Update Policy

## Main Hub vs Watchlist

Use the main hub for resources with a clear use case and enough information to guide another engineer. Use watchlist for resources that are promising but not yet verified.

Do not store full external repositories in this hub. Use temporary clones for inspection, then keep only curated entries, links, metadata, and internal notes.

## Index Updates

Update `INDEX.md` only when:

- adding a new major category
- adding an important internal skill
- adding a case that should be discoverable from the top level
- changing the hub structure

Do not add every small resource to `INDEX.md`; category pages should carry the detail.

## Document Growth

Growth is expected, but keep it folded into leaf pages:

- Top-level pages should orient readers.
- Category pages should support quick comparison.
- Detail pages should hold full notes, risks, verification, and update history.

When a category entry grows beyond roughly 10-12 lines, move the long content to a detail page and keep only a quick card in the category page.

## Experience Notes

Experience notes should preserve chronology:

- Write the original observation under the original date.
- Add follow-up notes under `Later Updates`.
- Link reusable conclusions back to a resource, case, or skill when possible.

## Review Rhythm

Suggested `Last checked` cadence:

- High-priority tools: every 1-2 months.
- Medium-priority tools: every quarter.
- Watchlist items: set an explicit recheck date.
- Deprecated items: only revisit when someone asks about them.
